¿Qué es Git y por qué salva vidas?

Imagina este escenario real con tu equipo...

Escenario sin Git (el caos)

María hace cambios en app.py y lo manda por WhatsApp. Juan también hizo cambios en app.py y los mandó por correo. Pedro mezcla ambos archivos a mano, borra algo sin querer, y nadie sabe qué versión era la buena. Son las 2 AM y el proyecto se entrega mañana. 😱

Con Git (el orden)

Cada integrante trabaja en su propia rama. Los cambios se guardan con historial completo. Si algo se rompe, se revierte en segundos. El equipo fusiona el trabajo limpiamente. ✅

Definición concreta

Git es un sistema de control de versiones distribuido. Registra cada cambio que hacés en tu código, quién lo hizo y cuándo. Es como un "Ctrl+Z" infinito y colaborativo para tu proyecto.

GitHub es la nube donde vive el repositorio. Git corre en tu computadora; GitHub es el servidor compartido.

Conceptos clave
  • Repositorio (repo): La carpeta de tu proyecto bajo control de Git. Contiene todo el historial.
  • Commit: Una "foto" del estado de tu código en un momento dado. Cada commit tiene un mensaje descriptivo.
  • Rama (branch): Una línea de desarrollo paralela. Permite trabajar sin afectar al resto del equipo.
  • Merge: La acción de fusionar el trabajo de una rama con otra.
Instalación & Configuración Inicial

Hacé esto una sola vez en tu computadora. Luego nunca más lo vas a necesitar.

Paso 1: Instalar Git

Windows

Descargá el instalador desde git-scm.com. Durante la instalación, asegurate de seleccionar "Git Bash" y "Use Git from the command line and also from 3rd-party software".

macOS
Terminal — macOS
# Instalá Homebrew primero si no lo tenés:
$ /bin/bash -c "$(curl -fsSL https://brew.sh/install.sh)"

# Luego instalá Git:
$ brew install git
Linux (Ubuntu/Debian)
Terminal — Linux
$ sudo apt update && sudo apt install git -y
Verificar instalación
$ git --version
git version 2.43.0

Paso 2: Crear cuenta en GitHub

  • 1
    Andá a github.com y hacé clic en "Sign up".
  • 2
    Elegí un nombre de usuario profesional (lo va a ver tu futuro empleador). Usá algo como juan-garcia-ds.
  • 3
    Verificá tu correo electrónico.
  • 4
    Explorá tu perfil y empezá a construir tu portafolio. ¡Cada commit cuenta!

Paso 3: Configurar Git con tu identidad

Esto le dice a Git quién sos vos. Hacé esto exactamente una vez en tu computadora.

git config — Configuración global
# Reemplazá los datos con los tuyos:

$ git config --global user.name "Juan García"
$ git config --global user.email "juan@email.com"

# Configurá el editor por defecto (VS Code recomendado):
$ git config --global core.editor "code --wait"

# En Windows: establecé el salto de línea (evita bugs de compatibilidad):
$ git config --global core.autocrlf true

# Verificar que todo quedó bien:
$ git config --list
user.name=Juan García
user.email=juan@email.com
core.editor=code --wait
Regla de Oro #1

Usá el mismo email con el que creaste tu cuenta de GitHub. Git usa este email para asociar tus commits a tu perfil. Si usás uno diferente, tus contribuciones no van a aparecer en tu gráfico de actividad de GitHub.

El archivo .gitignore

El guardián de tu repositorio. Decide qué no debe subirse a la nube.

¿Por qué es vital para proyectos Flask?

En proyectos Python/Flask se generan automáticamente carpetas y archivos que nunca deben subirse a GitHub porque:

  • Son específicos de tu computadora y no van a funcionar en la del compañero.
  • Pueden pesar cientos de megabytes (el entorno virtual).
  • Pueden contener contraseñas y claves secretas (¡el error más grave!).
  • Ensucian el historial del repositorio con archivos irrelevantes.

.gitignore para proyectos Flask

Creá este archivo en la raíz de tu proyecto (al mismo nivel que app.py):

.gitignore
# ═══ Entorno Virtual de Python ═══
# NUNCA subir el venv. Ocupa cientos de MB.
# Cada quien instala el suyo con: pip install -r requirements.txt
venv/
.venv/
env/
ENV/

# ═══ Caché de Python ═══
# Python crea estos archivos automáticamente al ejecutar el código
__pycache__/
*.py[cod]
*.pyo
*.pyc

# ═══ Variables de entorno y secretos ═══
# CRÍTICO: aquí van contraseñas, API keys, etc.
.env
.env.local
*.env

# ═══ Base de datos local ═══
*.db
*.sqlite3

# ═══ Sistema operativo ═══
.DS_Store    # macOS
Thumbs.db    # Windows

# ═══ IDEs y editores ═══
.vscode/
.idea/
*.swp
Regla de Oro #2 — La más importante

Jamás subas archivos .env a GitHub. Esos archivos contienen contraseñas, claves de API y tokens de bases de datos. Si lo subís aunque sea por un segundo, considerá esa clave comprometida. Hay bots que escanean GitHub 24/7 buscando exactamente eso. Siempre agregá .env al .gitignore antes de hacer el primer commit.

requirements.txt — El sustituto del venv

En lugar de subir el entorno virtual, compartís un archivo con la lista de dependencias:

Generar y usar requirements.txt
# Vos (líder del proyecto) generás el archivo:
$ pip freeze > requirements.txt

# Subís requirements.txt a GitHub (NO el venv).

# Tus compañeros, luego de hacer git clone, ejecutan:
$ python -m venv venv    # Crean su propio entorno virtual
$ source venv/bin/activate  # Linux/Mac
$ venv\Scripts\activate    # Windows
(venv) $ pip install -r requirements.txt
Successfully installed Flask-3.0.0 Werkzeug-3.0.1 ...
Crear y Conectar Repositorios

Hay dos maneras de empezar. La que uses depende de si ya existe el proyecto o no.

Opción A: Empezar desde cero (git init)

Usás esta opción cuando el proyecto todavía no existe en ningún lado. La persona líder del equipo hace esto.

  • 1
    Crear el repositorio en GitHub primero
    Andá a github.com → botón verde "New" → Poné el nombre del repo (ej: proyecto-flask-equipo) → NO inicialices con README → Create repository.
  • 2
    Inicializar el repo local y conectarlo
    git init + remote add
    # Entrá a la carpeta de tu proyecto:
    $ cd mi-proyecto-flask

    # Inicializá Git en esa carpeta (crea la carpeta .git oculta):
    $ git init
    Initialized empty Git repository in /mi-proyecto-flask/.git/

    # Creá el .gitignore ANTES de hacer cualquier commit:
    $ touch .gitignore  # (luego completalo con el contenido de la sección anterior)

    # Conectar con el repositorio de GitHub (copiá la URL del repo):
    $ git remote add origin https://github.com/tu-usuario/proyecto-flask-equipo.git

    # Verificar que la conexión quedó bien:
    $ git remote -v
    origin https://github.com/tu-usuario/proyecto-flask-equipo.git (fetch)
    origin https://github.com/tu-usuario/proyecto-flask-equipo.git (push)

Opción B: Clonar un repo existente (git clone)

Usás esta opción cuando el proyecto ya existe en GitHub. Tus compañeros hacen esto para unirse al proyecto.

git clone — Descargar un repositorio existente
# Copiá la URL del repositorio desde GitHub (botón verde "Code")
$ git clone https://github.com/lider/proyecto-flask-equipo.git
Cloning into 'proyecto-flask-equipo'...
remote: Enumerating objects: 42, done.
Receiving objects: 100% (42/42), 15.23 KiB | 2.18 MiB/s, done.

# Entrá a la carpeta descargada:
$ cd proyecto-flask-equipo

# Verificá que ya está conectado a GitHub automáticamente:
$ git remote -v
origin https://github.com/lider/proyecto-flask-equipo.git (fetch)
origin https://github.com/lider/proyecto-flask-equipo.git (push)
¿Cuál usar en el proyecto grupal?

Persona 1 (Líder): Crea el repo en GitHub → lo clona o usa git init + remote add → hace el primer commit con la estructura básica.
Personas 2 y 3: Usan git clone con la URL que les pasa el líder. ¡Así de simple!

Staging Area & Commits

El corazón de Git. Acá es donde guardás tus avances de forma permanente.

¿Qué es el Staging Area?

Pensalo como una zona de preparación. Es el lugar donde ponés los cambios que querés incluir en tu próxima "foto" (commit). No todos los archivos modificados tienen que ir en el mismo commit.

git status — Ver qué está pasando

Tu comando favorito. Usalo constantemente para saber en qué estado está tu repositorio.

git status — El diagnóstico
$ git status

On branch feature/login
Your branch is up to date with 'origin/feature/login'.

Changes not staged for commit:
(use "git add <file>..." to update what will be committed)
modified: app.py
modified: templates/login.html

Untracked files:
(use "git add <file>..." to include in what will be committed)
static/css/styles.css

git add — Preparar cambios

git add — Variantes
# Agregar un archivo específico (recomendado para commits precisos):
$ git add app.py

# Agregar múltiples archivos específicos:
$ git add app.py templates/login.html

# Agregar TODOS los cambios en el directorio actual:
$ git add .

# Verificar que quedaron en staging (aparecen en verde):
$ git status
Changes to be committed:
(use "git restore --staged <file>..." to unstage)
modified: app.py
modified: templates/login.html

git commit — Guardar los cambios

Un commit es permanente en el historial. El mensaje que escribís es crucial.

git commit
# La forma más común (mensaje corto):
$ git commit -m "feat: agregar formulario de login con validación"
[feature/login 3f4a2b1] feat: agregar formulario de login con validación
2 files changed, 47 insertions(+), 3 deletions(-)
Cómo escribir buenos mensajes de commit

Una convención muy usada es el Conventional Commits:

PrefijoCuándo usarloEjemplo
feat: Nueva funcionalidad feat: agregar endpoint de usuarios
fix: Corrección de un bug fix: corregir error 404 en /login
docs: Solo documentación docs: actualizar README con instrucciones
style: Formato, CSS, sin lógica style: centrar navbar en mobile
refactor: Reestructurar código refactor: extraer lógica de DB a models.py
Mensajes de commit que NO deben existir

❌ "cambios"  ❌ "arreglé cosas"  ❌ "asdfjkl"  ❌ "no sé qué hice"
Un buen commit message es como dejarle una nota a tu yo del futuro (o a tu compañero).

Push, Pull & Historial

Sincronizá tu trabajo con GitHub y mantené tu repositorio actualizado.

git push — Subir a la nube

Una vez que hiciste tu commit local, subís esos cambios a GitHub para que los demás los puedan ver.

git push
# Subir la rama actual a GitHub:
$ git push origin feature/login
Enumerating objects: 7, done.
Counting objects: 100% (7/7), done.
Writing objects: 100% (4/4), 1.23 KiB | 1.23 MiB/s, done.
To https://github.com/usuario/proyecto.git
abc1234..3f4a2b1 feature/login -> feature/login

# Primera vez que subís una nueva rama (necesita -u para trackearla):
$ git push -u origin feature/login
# Después de esto, podés usar solo "git push" sin argumentos

git pull — Traer cambios de compañeros

Antes de empezar a trabajar cada día, siempre actualizá tu código local con los cambios de tus compañeros.

git pull — Actualizar desde GitHub
# Descargar y aplicar los cambios de GitHub:
$ git pull origin main
remote: Enumerating objects: 5, done.
Updating abc1234..9e2f341
Fast-forward
templates/index.html | 12 ++++++------
1 file changed, 6 insertions(+), 6 deletions(-)
Regla de Oro #3 — El ritual diario

Cada vez que te sentás a trabajar, lo primero es:
1. git pull origin main → actualizar la rama principal
2. git checkout tu-rama → pasarte a tu rama
3. git merge main → traer los últimos cambios a tu rama
Esto evita el 90% de los conflictos molestos.

Ramas (Branches)

El superpoder de Git para equipos. Cada integrante trabaja en su propia línea sin pisarse.

Regla de Oro #4 — La más importante del equipo

NADIE trabaja directamente en la rama main (o master). Jamás. La rama main es la versión "oficial" y funcional del proyecto. Cada persona crea su propia rama para desarrollar. Solo se fusiona a main cuando el código está revisado y aprobado.

Comandos de ramas

git branch — Gestión de ramas
# Ver todas las ramas (la activa tiene un *):
$ git branch
feature/dashboard
* feature/login
main

# Ver también las ramas remotas (en GitHub):
$ git branch -a

# Crear una nueva rama:
$ git branch feature/nueva-funcionalidad

# Cambiar a otra rama:
$ git checkout feature/nueva-funcionalidad
Switched to branch 'feature/nueva-funcionalidad'

# ⭐ El atajo más usado: CREAR y CAMBIAR a la vez:
$ git checkout -b feature/nueva-funcionalidad
Switched to a new branch 'feature/nueva-funcionalidad'

Convenciones para nombrar ramas

Usa un formato consistente para que el equipo entienda qué hace cada rama sin leerla:

TipoFormatoEjemplo real del proyecto
Nueva función feature/descripcion feature/login-usuarios
Corrección fix/descripcion fix/error-404-dashboard
Hotfix urgente hotfix/descripcion hotfix/crash-al-guardar
Por persona juan/descripcion juan/modulo-graficos
Buenas prácticas de naming
  • Usá minúsculas y guiones: feature/login-usuarios ✅   Feature/LoginUsuarios ❌
  • Sé descriptivo pero conciso: feature/endpoint-prediccion-ml ✅   mi-rama ❌
  • Sin espacios ni caracteres especiales.
Merge & Resolución de Conflictos

Fusionando el trabajo del equipo y sobreviviendo a los conflictos.

git merge — Fusionar ramas

Cuando terminás tu funcionalidad, la integrás a main. El proceso es siempre el mismo:

  • 1
    Ir a la rama destino (main) y actualizarla
    git merge — Paso a paso
    # 1. Ir a la rama principal:
    $ git checkout main

    # 2. Traer lo último de GitHub:
    $ git pull origin main

    # 3. Fusionar tu rama con main:
    $ git merge feature/login
    Updating 9e2f341..3f4a2b1
    Fast-forward
    app.py | 15 +++++++++++++++
    templates/login.html | 42 ++++++++++++++++++++++++++++++++++++++++++
    2 files changed, 57 insertions(+)

    # 4. Subir el main actualizado:
    $ git push origin main

Conflictos de Merge — ¡No entres en pánico!

¿Por qué ocurre un conflicto?

Un conflicto ocurre cuando dos personas modificaron exactamente la misma línea del mismo archivo. Git no sabe cuál versión es la "correcta", entonces te pide que vos lo decidas manualmente.
Por ejemplo: Juan editó la línea 15 de app.py y María también editó la línea 15 de app.py. Cuando se intenta hacer merge → 💥 conflicto.

¿Cómo se ve un conflicto?

Git modifica el archivo afectado y agrega marcadores especiales:

⚠ CONFLICTO DETECTADO — app.py — línea 15
...

<<<<<<< HEAD
# Esta es TU versión (la que está en la rama actual, ej: main)
app.run(debug=True, host='0.0.0.0', port=5000)
=======
# Esta es la versión de tu compañero (la rama que estás mergeando)
app.run(debug=False, port=8080)
>>>>>>> feature/config-servidor

...
Leyendo los marcadores
  • <<<<<<< HEAD → Empieza TU versión (la rama donde estás parado actualmente).
  • ======= → Separador entre las dos versiones.
  • >>>>>>> feature/config-servidor → Empieza la versión de la otra rama.

Cómo resolver el conflicto — paso a paso

  • 1
    Abrí el archivo conflictivo en tu editor (VS Code lo resalta en rojo/verde automáticamente). Hablá con tu compañero y decidí juntos cuál es la versión correcta.
  • 2
    Editá el archivo manualmente. Borrá los marcadores y dejá solo el código correcto. Por ejemplo, si deciden combinar ambos:
    app.py — Después de resolver
    # Borraste los marcadores, quedó el código correcto:
    app.run(debug=True, host='0.0.0.0', port=5000)
  • 3
    Guardá el archivo y hacé git add para marcar el conflicto como resuelto:
    Finalizar la resolución del conflicto
    # Marcar el conflicto como resuelto:
    $ git add app.py

    # Confirmar el merge con un commit:
    $ git commit -m "merge: resolver conflicto en app.py — mantener config debug=True"
    [main 7c1a3f2] merge: resolver conflicto en app.py

    # Subir el resultado a GitHub:
    $ git push origin main
Cómo minimizar conflictos
  • Dividí bien el trabajo: Cada persona se encarga de archivos o módulos diferentes.
  • Hacé pull frecuentemente: Al inicio de cada sesión de trabajo.
  • Committeá seguido: Cambios pequeños y frecuentes evitan divergencias grandes.
  • Comunicate: "Oye, voy a tocar models.py, ¿alguien lo está usando?"
Herramientas GUI — Git con botones

Dominá la línea de comandos primero, pero sabé que existen herramientas visuales que facilitan mucho el trabajo.

¿Por qué aprender la terminal primero?

Las herramientas GUI son excelentes, pero si solo las usás a ellas, no entendés qué está pasando por debajo. Cuando algo falla (y va a fallar), vas a necesitar saber los comandos. Una vez que los dominás, las GUIs te hacen un 2x más rápido.
